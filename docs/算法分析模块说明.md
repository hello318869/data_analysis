# 算法分析模块说明

## 一、模块名称

算法分析模块。

本模块是交互式数据分析系统中的机器学习部分，主要完成“选择字段 → 训练模型 → 预测评估 → 图表展示 → 历史记录保存”的流程。我的工作重点不是重写整个系统，而是在已有 FastAPI、Jinja2、SQLAlchemy 项目结构上补齐算法分析功能。

## 二、模块职责

算法分析模块主要负责以下内容：

- 接收用户在页面上选择的特征列 `features` 和目标列 `target`。
- 使用线性回归进行模型训练、预测和评估。
- 支持 `LinearRegression`、`Ridge`、`Lasso` 三种回归算法对比。
- 生成“预测值 vs 真实值”散点图，辅助观察预测效果。
- 登录用户完成分析后，将分析参数、指标摘要和图表路径保存到历史记录表。

## 三、核心文件说明

### 1. `services/ml_service.py`

这是算法模块的业务逻辑文件，不直接接触 FastAPI 的 `request` 或 `session`。

主要功能包括：

- 校验特征列、目标列是否存在以及是否为数值类型。
- 只删除 `features + target` 相关列中的缺失值行。
- 按 80/20 划分训练集和测试集。
- 训练 `LinearRegression` 模型，返回 R²、MSE、MAE、回归系数、截距、预测明细等结果。
- 对比 `LinearRegression`、`Ridge`、`Lasso` 三种模型，并根据 R² 最大、MSE 更小的规则选出最佳算法。
- 使用 Matplotlib 生成散点图，并保存到 `outputs/charts/`。
- 通过 `font_manager` 检测系统字体，避免中文字体缺失时出现大量 glyph warning。

### 2. `routes/analysis_routes.py`

这是算法模块的路由文件，负责把 Web 请求和算法服务连接起来。

主要接口：

- `GET /analysis`：展示算法分析页面。
- `POST /analysis/regression`：执行线性回归分析。
- `POST /analysis/compare`：执行三种算法对比。
- `GET /analysis/result`：查看 session 中保存的上一次分析结果。

路由层会从 `session["df_json"]` 读取数据，还原为 DataFrame，然后调用 `services/ml_service.py` 中的函数。分析成功后会把结果保存到 `session["last_analysis"]`，登录用户还会写入 `AnalysisRecord`。

### 3. `templates/analysis.html`

这是算法分析页面模板，继承项目已有的 `base.html`。

页面包含：

- 当前文件名、字段数量、数值字段列表。
- 线性回归分析表单。
- 算法对比表单。
- R²、MSE、MAE 指标卡片。
- 训练集和测试集大小。
- 特征列、目标列、截距。
- 回归系数表。
- 最多前 20 行预测结果表。
- “预测值 vs 真实值”散点图。
- 算法对比表，并高亮最佳算法。

表单提交时使用 `fetch` 发送 JSON，保证 `Content-Type: application/json`，符合接口契约。

### 4. `tests/test_ml_service.py`

这是用于手动测试算法服务的脚本，可以直接运行：

```bash
python tests/test_ml_service.py
```

脚本会读取 `data/sample_housing.csv`，使用：

```python
features = ["sqft", "bedrooms", "bathrooms", "age"]
target = "price"
```

然后先对涉及列执行 `dropna`，再调用 `run_linear_regression` 和 `compare_regression_models`，最后用 `assert` 检查返回结果是否基本正确。

### 5. `main.py`

`main.py` 中主要有两处和算法模块相关的改动。

第一处是注册算法路由：

```python
from routes import analysis_routes

app.include_router(analysis_routes.router, prefix="/analysis", tags=["算法分析"])
analysis_routes.templates = templates
```

第二处是数据库初始化容错。原本项目启动时会直接执行 `Base.metadata.create_all(bind=engine)`。在本机环境中，默认 MySQL 账号 `root/root` 认证失败会导致 `uvicorn` 无法启动。为了不影响没有配置好数据库时的页面联调，增加了容错：

```python
try:
    Base.metadata.create_all(bind=engine)
except SQLAlchemyError as exc:
    print(f"数据库初始化失败，应用将继续启动：{exc}")
```

这样做不会删除数据库初始化逻辑，只是避免数据库暂时不可用时整个 Web 应用直接启动失败。

## 四、算法流程

算法分析的整体流程如下：

1. 用户进入 `/analysis` 页面。
2. 路由从 `session["df_json"]` 中读取数据，并还原为 Pandas DataFrame。
3. 页面展示所有字段和数值字段，用户选择特征列和目标列。
4. 提交 `/analysis/regression` 或 `/analysis/compare`。
5. 后端校验字段是否存在、是否为数值类型。
6. 只删除特征列和目标列中的缺失值行，不处理无关列。
7. 使用 `train_test_split` 按 80/20 划分训练集和测试集，`random_state=42`。
8. 训练模型。
9. 对测试集进行预测。
10. 计算 R²、MSE、MAE。
11. 生成“预测值 vs 真实值”散点图。
12. 将结果传回 `analysis.html` 展示。
13. 如果用户已登录，把分析记录写入 `analysis_records` 表。

## 五、关键代码片段

### 1. 数据校验

```python
def validate_regression_columns(df, features, target):
    if not features:
        raise ValueError("请至少选择 1 个特征列")

    if target not in df.columns:
        raise ValueError(f"目标列 {target} 不存在")

    missing_features = [feature for feature in features if feature not in df.columns]
    if missing_features:
        raise ValueError(f"特征列 {'、'.join(missing_features)} 不存在")

    if not is_numeric_dtype(df[target]):
        raise ValueError(f"目标列 {target} 不是数值类型，无法用于回归")

    for feature in features:
        if not is_numeric_dtype(df[feature]):
            raise ValueError(f"特征列 {feature} 不是数值类型，无法用于回归")
```

这部分主要防止用户把文本列、分类列或不存在的列拿去训练模型。

### 2. 只删除相关列缺失值

```python
regression_columns = features + [target]
clean_df = df.loc[:, regression_columns].dropna(subset=regression_columns).copy()
```

这里没有对整个 DataFrame 执行 `dropna()`，因为无关列的缺失值不应该影响回归训练。

### 3. 划分训练集和测试集

```python
x_train, x_test, y_train, y_test = train_test_split(
    x_data,
    y_data,
    test_size=0.2,
    random_state=42,
)
```

这样可以保证训练集和测试集比例为 80/20，同时每次运行结果可复现。

### 4. LinearRegression 训练

```python
model = LinearRegression()
model.fit(x_train, y_train)
predicted = model.predict(x_test)
```

训练完成后，会把预测值和真实值组成表格，返回给页面展示。

### 5. Ridge / Lasso 对比

```python
models = {
    "LinearRegression": LinearRegression(),
    "Ridge": Ridge(),
    "Lasso": Lasso(max_iter=10000),
}

for algorithm, model in models.items():
    model.fit(x_train, y_train)
    predicted = model.predict(x_test)
    metrics = _calculate_metrics(y_test, predicted)
```

三种算法使用同一份训练集和测试集，保证对比比较公平。

### 6. 指标计算

```python
{
    "r2_score": float(r2_score(actual, predicted)),
    "mse": float(mean_squared_error(actual, predicted)),
    "mae": float(mean_absolute_error(actual, predicted)),
}
```

这里统一转成 Python 原生 `float`，避免后面保存到 session 或数据库 JSON 字段时出现 numpy 类型不能序列化的问题。

### 7. Matplotlib 图表生成

```python
fig, ax = plt.subplots(figsize=(7, 5), dpi=120)
try:
    ax.scatter(actual_values, predicted_values, alpha=0.75, color="#2563eb")
    ax.plot(
        [min_value, max_value],
        [min_value, max_value],
        linestyle="--",
        color="#dc2626",
        linewidth=1.5,
        label=reference_label,

    fig.savefig(file_path, bbox_inches="tight")
```

使用 `finally` 关闭 figure，可以避免 Web 服务长时间运行后图表对象堆积。

### 8. 字体兼容处理

```python
available_fonts = {font.name for font in font_manager.fontManager.ttflist}

for font_name in FONT_PRIORITY:
    if font_name in available_fonts:
```

如果找到中文字体，就使用中文标题和坐标轴；如果只找到 `DejaVu Sans` 或没有中文字体，就自动切换成英文图表文案，避免终端出现大量 glyph warning。

## 六、遇到的问题及解决

### 1. 非数值列不能参与回归

一开始如果用户选择了文本列，scikit-learn 会在训练时报错。解决方法是在训练前使用 `is_numeric_dtype` 做字段类型校验，发现非数值列就直接返回友好的错误提示。

### 2. 缺失值导致模型无法训练

示例数据中有一些空值，如果直接训练会报错。解决方法是只对 `features + target` 这些相关列执行 `dropna`。这样既能保证模型可训练，也不会因为无关列缺失而误删数据。

### 3. numpy 类型不能直接写入 session / JSON

模型指标和预测结果中会出现 `numpy.float64`、`numpy.int64`。这些类型在写入 session 或数据库 JSON 字段时可能无法序列化。解决方法是在 service 层统一转为 Python 原生 `float`、`int`、`list`、`dict`。

### 4. Matplotlib Web 环境需要 Agg 后端

Web 服务环境通常没有图形界面，如果 Matplotlib 使用默认交互式后端，可能会报错。解决方法是在 service 文件中设置：

```python
matplotlib.use("Agg")
```

这样可以在服务端直接生成 PNG 文件。

### 5. 当前环境缺少中文字体导致 glyph warning

本机环境没有合适的中文字体时，Matplotlib 会输出大量 glyph warning。解决方法是通过 `matplotlib.font_manager` 检测系统字体，找到中文字体就使用中文，否则自动改成英文标题和坐标轴。

### 6. MySQL root/root 认证失败导致 uvicorn 启动阻塞

项目默认数据库连接是 `root/root`，但当前本机 MySQL 不接受这个账号密码，导致 `uvicorn main:app --reload` 在启动阶段失败。解决方法是在 `main.py` 中给数据库初始化加容错，数据库不可用时打印警告，但不影响 Web 页面和算法功能联调。

## 八、收获与改进

通过这个模块，我对机器学习功能如何接入 Web 系统有了比较完整的理解。以前只是在 notebook 或单独脚本里训练模型，这次需要考虑用户输入、字段校验、页面展示、session 保存、图表路径、数据库历史记录等问题，感觉更接近一个真正的小型系统。

这次也加深了我对 FastAPI 路由、Jinja2 模板、session 和 SQLAlchemy 历史记录之间协作方式的理解。算法服务本身不直接操作 request/session，路由层负责 Web 逻辑，这样代码边界会更清楚。

后续如果继续扩展，可以考虑加入：

- 决策树回归、随机森林回归等更多模型。
- 特征标准化和归一化选项。
- 分类变量自动编码。
- 训练参数可配置，例如 Ridge/Lasso 的 alpha。
- 更详细的模型解释，比如残差分析、特征重要性等。
